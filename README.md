# AISapp

Продуктовая локальная версия информационной системы для учета материалов, закупок, складских операций и связанного документооборота в строительно-монтажной организации.

Документ описывает текущую поставку проекта по состоянию на `06.05.2026` и предназначен для:

- локального развертывания системы;
- демонстрации готового функционала;
- запуска демо-стенда с тестовыми данными;
- дальнейшего рефакторинга и сопровождения продукта.

## 1. Что это за продукт

`AISapp` объединяет в одном веб-приложении:

- договорной контур;
- закупочный контур;
- складской учет;
- списание материалов по нормам;
- учет выдачи СИЗ и спецодежды;
- журнал работ по участкам;
- архив и жизненный цикл документов;
- отчетность;
- резервное копирование и восстановление;
- аудит действий пользователей;
- REST API для ключевых сущностей.

Текущая поставка представляет собой монолитное Django-приложение с PostgreSQL в качестве основной БД и подготовленной интеграцией с Redis/Celery. Операционные табличные позиции хранятся в едином JSON-формате.

## 2. Продуктовый профиль

Тип поставки:
- локальная продуктовая сборка для внутреннего использования, пилотного внедрения, демонстрации и безопасного рефакторинга.

Целевая предметная область:
- строительные и электромонтажные организации;
- учет материалов и документов по объектам и участкам;
- контроль движения материалов от закупки до списания или выдачи сотруднику.

Ключевая ценность:
- единый контур учета документов и материалов;
- сквозная роль-ориентированная работа;
- управляемый архив документов;
- быстрый демонстрационный запуск с готовыми данными.

## 3. Основной функционал

### 3.1. Справочники

В системе реализованы справочники:

- материалы;
- поставщики;
- строительные объекты;
- работники;
- нормы расхода материалов;
- договоры СМР;
- договоры поставки;
- типы документов;
- пользователи.

Для пользовательских справочников реализован CRUD через UI:

- создание;
- просмотр;
- редактирование;
- деактивация;
- удаление там, где это не блокируется связанными данными.

### 3.2. Операционный контур

В системе поддерживаются операции:

- заявки на закупку;
- заявки начальника участка на материалы со склада;
- загрузка документов поставщика;
- формирование первичных документов;
- приход материалов на центральный склад;
- отпуск материалов на участок;
- акты списания;
- акты сдачи-приемки выполненных работ;
- выдача СИЗ и спецодежды;
- журнал работ по участкам.

### 3.3. Документооборот и архив

Поддерживается жизненный цикл документов со статусами:

- `draft`;
- `approval`;
- `approved`;
- `sent_accounting`;
- `accepted`;
- `rework`;
- `uploaded`;
- `supply_confirmed`.

Поддерживается:

- согласование по ролям;
- возврат на доработку;
- передача в бухгалтерию;
- завершение бухгалтерского этапа;
- единый архив документов;
- фильтрация архива по типу, номеру, статусу, датам, контрагенту, объекту и поисковой строке.

### 3.4. Складской учет

Реализовано:

- ведение движений материалов;
- центральный склад;
- остатки по участкам;
- контроль минимальных остатков;
- защита от отрицательных остатков;
- отражение списания и выдачи СИЗ в движениях.

### 3.5. Отчеты

В системе доступны отчеты:

- остатки материалов за период;
- материальный отчет участка за период;
- закупки материалов;
- списания материалов по объектам;
- работа участков;
- сводный отчет для бухгалтерии;
- выданная спецодежда и сроки службы;
- движение материалов;
- статистика по расходу материалов;
- статистика по выполненным работам.

Поддерживается экспорт отчетов в `XLSX`.

### 3.6. Экспорт документов

Поддерживается экспорт документов в `DOCX` и экспорт отчетов в `XLSX`.

Документы формируются на основе проектных шаблонов из папки `document_templates/`. В шаблонах используются placeholders вида `{{FIELD_NAME}}`; при экспорте система подставляет реквизиты договора, заявки, поставщика, объекта, строк материалов, СИЗ и подписантов. Если нужно временно использовать другой каталог шаблонов, задайте `AIS_DOCUMENT_TEMPLATES_DIR` в `.env`.

### 3.7. Backup и аудит

Поддерживается:

- создание резервных копий из UI;
- восстановление из JSON backup через UI;
- восстановление из JSON backup через management command;
- журнал действий пользователей в отдельном интерфейсе.

## 4. Базовые роли в проекте

В проект заложены следующие роли:

- `admin`
- `director`
- `procurement`
- `warehouse`
- `site_manager`
- `accounting`
- `supplier`

Ниже приведена рабочая матрица ролей по текущей продуктовой конфигурации.

| Роль | Назначение | Основной доступ |
| --- | --- | --- |
| `admin` | Полное администрирование продукта | Все справочники, все операции, архив, отчеты, резервные копии, журнал действий, управление пользователями |
| `director` | Руководитель / согласующий контур | Справочники офисного контура, договоры СМР, договоры поставки, архив, отчеты, дашборд |
| `procurement` | Снабжение | Справочники офисного контура, договоры поставки, заявки участка, заявки поставщику, документы поставщиков, первичные документы, приходы, архив, отчеты |
| `warehouse` | Склад | Справочники офисного контура, заявки участка, первичные документы, приходы, отпуски, выдача СИЗ, архив, отчеты |
| `site_manager` | Начальник участка | Материалы, поставщики, объекты, работники, нормы, договоры СМР в read-only и только по своему контуру, заявки кладовщику, акты списания, акты сдачи-приемки, выдача СИЗ, журнал работ, архив, отчеты по своему участку |
| `accounting` | Бухгалтерия | Дашборд, архив, отчеты, завершение бухгалтерского этапа документа в архиве |
| `supplier` | Внешний поставщик | Просмотр своих договоров поставки, просмотр своих заявок, загрузка своих документов поставщика, архив по своим документам |

Дополнительно:

- только `admin` имеет доступ к резервным копиям;
- только `admin` имеет доступ к журналу действий;
- только `admin` управляет пользователями через UI;
- роль `supplier` не имеет доступа к разделу отчетов;
- роль `site_manager` не создает договоры СМР, а работает с уже доступным ему договорным контуром;
- роль `supplier` работает только в рамках связанного поставщика.

## 9. Локальное развертывание на Windows

### Шаг 1. Откройте проект

```powershell
cd "C:\Путь к проекту\AISapp"
```

### Шаг 2. Создайте виртуальное окружение

```powershell
python -m venv .venv
```

### Шаг 3. Активируйте виртуальное окружение

```powershell
& .\.venv\Scripts\Activate.ps1
```

### Шаг 4. Установите зависимости

```powershell
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

### Шаг 5. Подготовьте `.env`

```powershell
Copy-Item .env.example .env
```

После этого при необходимости отредактируйте `.env` под вашу локальную БД и окружение.

### Шаг 6. Убедитесь, что PostgreSQL доступен

По умолчанию проект ожидает:

```text
host=localhost port=5433 dbname=ais_db user=postgres password=root
```

Важно:

- `manage.py` автоматически пытается создать БД `ais_db`, если она отсутствует;
- для этого пользователь PostgreSQL должен иметь право подключения к БД `postgres` и право `CREATE DATABASE`;
- если таких прав нет, создайте БД `ais_db` вручную и затем выполните миграции.

### Шаг 7. Выполните миграции

```powershell
python manage.py migrate --noinput
```

### Шаг 8. Создайте продуктового администратора

Вариант 1. Явной командой:

```powershell
python manage.py bootstrap_product --username owner --password "StrongPassword123!" --email owner@example.com
```

Вариант 2. Через `.env`:

- заполните `AIS_INITIAL_ADMIN_USERNAME` и `AIS_INITIAL_ADMIN_PASSWORD`;
- затем запустите `python run.py`;
- `run.py` сам выполнит `bootstrap_product --from-env --noinput`.

### Шаг 9. При необходимости создайте базовые учетные записи по ролям

```powershell
python manage.py bootstrap_role_accounts
```

Команда создаёт или обновляет типовые аккаунты:

- `admin / Admin@AIS2026`
- `director / Director@AIS2026`
- `procurement / Procure@AIS2026`
- `warehouse / Warehouse@AIS2026`
- `site_manager / SiteMgr@AIS2026`
- `accounting / Accounting@AIS2026`
- `supplier / Supplier@AIS2026`

Если нужно принудительно перезаписать пароли:

```powershell
python manage.py bootstrap_role_accounts --reset-passwords
```

### Шаг 10. Запустите приложение

Предпочтительный способ локального запуска:

```powershell
python run.py
```

Что делает `run.py`:

- подгружает `.env`;
- выполняет `migrate --noinput`;
- при наличии `AIS_INITIAL_ADMIN_*` создаёт или обновляет стартового администратора;
- запускает сервер на `127.0.0.1:8010`.

Локальный URL:

```text
http://127.0.0.1:8010
```

Стандартные маршруты:

- `/` — пользовательский интерфейс;
- `/login/` — вход;
- `/admin/` — Django admin;
- `/api/` — REST API.

## 10. Развертывание через Docker Compose

В репозитории уже есть `Dockerfile` и `docker-compose.yml`.

### Быстрый запуск

```powershell
docker compose up --build
```

Поднимутся сервисы:

- `postgres`
- `redis`
- `web`
- `celery`

Параметры из `docker-compose.yml`:

- web/celery собираются из `python:3.12-slim`;
- PostgreSQL использует образ `postgres:15`;
- Redis использует образ `redis:7`;
- PostgreSQL контейнера публикуется как `5433 -> 5432`;
- веб-приложение публикуется на `8010`;
- Redis публикуется на `6379`.

После запуска можно создать продуктового администратора:

```powershell
docker compose exec web python manage.py bootstrap_product --username owner --password "StrongPassword123!" --email owner@example.com
```

Если нужны базовые ролевые аккаунты:

```powershell
docker compose exec web python manage.py bootstrap_role_accounts
```

Если нужен демо-набор:

```powershell
docker compose exec web python manage.py seed_demo_data --top-records 72 --prefix DEMO --replace
```

## 11. Демо-данные для демонстрации системы

Для демонстрации работоспособности и безопасного рефакторинга в проект добавлена команда генерации связного демо-набора:

```powershell
python manage.py seed_demo_data --top-records 72 --prefix DEMO --replace
```

Параметры команды:

- `--top-records` — количество верхнеуровневых операционных записей;
- `--prefix` — префикс данных и логинов;
- `--password` — общий пароль для demo-аккаунтов;
- `--replace` — пересоздать набор с тем же префиксом.

Ограничения команды:

- минимально `12` записей;
- максимально `100` записей;
- рекомендуемый диапазон для демонстрации: `50-100`.

## 12. Готовый backup демо-базы

В репозитории уже сохранён snapshot демонстрационного набора:

```text
backups/demo_seed_DEMO_20260421_183130.json
```

Это удобно, если нужно быстро восстановить демонстрационную БД без повторной генерации.

## 13. Восстановление из backup

Восстановление выполняется командой:

```powershell
python manage.py restore_backup ".\backups\demo_seed_DEMO_20260421_183130.json"
```

Важно:

- аргумент передается позиционно, без `--path`;
- восстановление заменяет текущие данные в прикладных таблицах на данные из backup;
- перед восстановлением рабочей БД рекомендуется сделать собственную резервную копию через UI.

В интерфейсе продукта восстановление также доступно в разделе `Резервные копии` для роли `admin`.

## 14. Создание резервной копии

- войдите под `admin`;

- создайте JSON backup;
- при необходимости скачайте его через интерфейс.

Папка хранения резервных копий:

## 18. Что важно знать сопровождающему разработчику

- `manage.py` умеет автоматически создать БД `ais_db`, если у пользователя PostgreSQL достаточно прав.
- `run.py` является основным локальным launcher-скриптом и сам выполняет миграции перед запуском.
- `run.py` включает `DJANGO_DEBUG=1`, если переменная не задана явно.
- Базовая бизнес-логика сосредоточена в `apps/core/services.py`, а отчетный слой вынесен в `apps/core/reporting.py`.
- Табличные позиции операций передаются в сервисы как JSON-массивы строк без отдельного текстового формата.
- Ролевая модель и доступ к интерфейсу завязаны на `apps/core/access.py`, `apps/core/views.py` и фильтрацию queryset по пользователю.
- Для рефакторинга безопаснее использовать `seed_demo_data` или готовый demo-backup, а не вручную собранную БД.

## 19. Статус текущей поставки

Текущая версия репозитория может использоваться как:

- локальный продуктовый стенд;
- демонстрационный контур для презентации;
- база для регрессионного тестирования;
- исходная точка для дальнейшего рефакторинга и промышленного усиления.

Если требуется полноценная production-эксплуатация, следующим шагом обычно становятся:

- удержание production baseline: `Python 3.12.x`, `PostgreSQL 15.x`, `Redis 7.x`;
- настройка внешнего reverse proxy;
- вынесение секретов и параметров окружения в защищенное хранилище;
- настройка постоянных томов для `backups`, `exports`, `uploads`, `media`;
- регламент резервного копирования и журналирования;
- CI/CD и автоматический smoke/regression прогон.
